# 第7章 系统实现

## 7.1 系统功能与平台架构

隧道工程安全监测涉及多源异构数据的实时采集、智能分析与协同管控，对系统平台的架构设计、技术选型和工程化实现提出了较高要求。基于前文所述的需求分析与系统设计方案，本章从系统实现角度，详细阐述隧道监测数字孪生平台的技术路线、核心模块实现与部署运行方案。本系统采用前后端分离的三层架构体系，前端基于React框架构建响应式交互界面，后端基于Flask框架提供RESTful API服务，数据层通过工厂模式实现Supabase云端数据库与MySQL本地数据库的灵活切换。系统整体覆盖数据采集与融合、智能分析与AI决策、集群控制与远程管控三大功能域，最终实现了一个面向隧道施工全生命周期的智能监测管理平台。

系统总体架构遵循经典的三层架构模式，自上而下依次为表现层、业务逻辑层和数据持久层，各层之间通过标准化接口进行松耦合通信。(1)表现层采用React 18与TypeScript构建单页应用（SPA），借助Vite 5实现模块热替换（HMR）与高效构建，通过Tailwind CSS原子化样式框架实现统一的视觉设计语言，并结合React Router v6实现15个核心页面的路由管理与代码分割懒加载，有效降低首屏加载体积。(2)业务逻辑层基于Flask 3.x框架搭建，核心入口文件api_server.py共计1258行，通过Blueprint机制将业务功能划分为15个独立模块，包括数据导入、预测分析、异常检测、工单管理、InSAR处理等，各模块独立注册路由前缀，实现了高内聚低耦合的后端服务架构。(3)数据持久层采用工厂模式设计，通过vendor.py中的DB_VENDOR环境变量实现数据源动态切换：当配置为"supabase"时，系统调用supabase_http_repo.py（775行）通过REST API访问Supabase云端PostgreSQL数据库；当配置为"mysql"时，系统调用mysql_repo.py（727行）通过本地驱动访问MySQL数据库。两种数据源实现统一的Repository接口，上层业务代码无需感知底层存储差异，体现了依赖倒置原则在工程中的实践。图7-1展示了系统总体架构的分层示意。

系统开发过程中的技术选型遵循成熟稳定、社区活跃、生态完善的原则，在保证开发效率的同时兼顾系统的可维护性与可扩展性。前端技术栈以React生态为核心，TypeScript提供静态类型检查以降低大规模代码库的维护风险，Vite替代传统Webpack实现亚秒级开发热更新；后端以Python语言为基础，Flask框架的轻量级特性便于与机器学习库（如scikit-learn、PyTorch）深度集成；数据库层面，Supabase提供免运维的云端PostgreSQL服务并内置行级安全策略，MySQL则作为离线部署或内网环境的备选方案。表7-1列出了系统的主要开发运行环境。

隧道监测系统的用户角色涵盖项目管理员、现场工程师和普通查看者，不同角色对系统功能的访问权限存在显著差异，因此认证授权机制是系统安全运行的基础保障。本系统实现了基于角色的访问控制（RBAC）模型，在AuthContext.tsx中定义了三级角色体系：(1)admin角色具备全部系统功能的访问权限，包括用户管理、模块配置和数据删除等敏感操作；(2)engineer角色可访问数据采集、分析预测、工单处理等核心业务功能；(3)viewer角色仅可查看监测数据与分析报告，无法执行写入操作。认证流程基于JWT（JSON Web Token）令牌机制实现，用户登录成功后系统签发包含角色信息的访问令牌，前端通过AuthContext全局状态管理令牌的存储与自动刷新。在路由层面，系统通过AuthGuard高阶组件实现页面级权限拦截，未认证用户自动重定向至登录页面，权限不足的用户则显示无权限提示。此外，系统支持通过VITE_AUTH_ENABLED环境变量全局开关认证功能，便于开发调试阶段快速绕过权限验证。

前端工程架构的设计以模块化、按需加载和动态可控为核心原则，旨在满足隧道监测系统功能多样且持续迭代的工程需求。(1)路由层面，系统基于React Router v6配置了15个核心页面路由，全部采用React.lazy()实现代码分割与懒加载，用户访问特定功能页面时才动态加载对应的JavaScript代码包，有效将首屏加载体积控制在合理范围内。(2)模块管理层面，系统设计了ModuleGate组件与ModulesContext上下文协同工作的动态模块管理机制。每个功能模块具有"developed"（已开发）和"pending"（待开发）两种状态，ModuleGate组件在渲染前检查目标模块的状态——已开发模块正常渲染功能界面，待开发模块则显示"开发中"占位提示。管理员可通过ModuleAdmin管理界面动态调整模块状态，且每次变更均记录审计字段（更新时间、操作人员），支持运行时的功能灰度发布与回滚。(3)状态管理层面，系统采用分层策略：全局认证状态通过AuthContext（React Context API）管理，模块配置通过ModulesContext管理，布局自适应通过LayoutContext管理（定义移动端、平板、桌面三个断点）；AI助手等高频交互状态则采用Zustand轻量状态库，agentStore中实现了30秒防抖机制以减少不必要的状态持久化写入；用户偏好设置通过localStorage实现跨会话持久化。

后端API服务的架构设计以模块化注册、统一异常处理和多级容错为核心特征，确保系统在复杂网络环境下的服务可用性。服务端入口api_server.py通过Flask Blueprint机制注册了15个业务模块，包括数据导入（data_import_bp）、预测分析（prediction_bp）、异常检测（anomaly_bp）、工单管理（tickets_bp）、InSAR处理（insar_bp）、盾构轨迹（shield_bp）等，每个Blueprint独立定义路由前缀与处理函数，支持团队并行开发与独立测试。前端与后端之间的通信链路采用三层API客户端架构实现渐进式降级：(1)基础层api.ts封装了统一的HTTP请求方法，针对服务端500错误实现1.5秒延迟自动重试机制；(2)中间层apiClient.ts在基础层之上增加了熔断与降级逻辑——当连续检测到2次请求失败后，系统自动切换至本地Mock数据模式，30秒冷却期结束后再尝试恢复真实API连接，避免因后端故障导致前端完全不可用；(3)专用层mlApi.ts针对机器学习预测接口进行定制化封装，处理模型推理特有的超时配置与响应格式解析。这一三层降级机制保障了系统在后端服务波动时仍能提供基本的数据展示功能，提升了面向现场用户的服务连续性。

系统的云端部署采用Vercel Serverless无服务器架构，通过vercel.json配置文件定义了关键运行参数：将Serverless函数的最大执行时间设置为60秒（maxDuration=60），以满足机器学习模型推理等计算密集型接口的响应需求；配置SPA路由重写规则，将所有非静态资源请求重定向至index.html，确保React Router的客户端路由在页面刷新和直接URL访问时正常工作。在移动端适配方面，系统基于Capacitor 6框架实现了Web应用向Android原生应用的跨平台封装。capacitor.config.ts中配置了Android WebView的服务地址与插件参数，前端通过VITE_MOBILE环境变量检测当前运行环境——当检测到移动端环境时，LayoutContext自动启用移动端布局模式，MobileCardSwitcher组件将桌面端的多栏并列布局转换为单栏卡片切换模式，确保在不同屏幕尺寸下均获得良好的交互体验。系统同时支持三个响应式断点：移动端（<768px）、平板端（768px~1024px）和桌面端（>1024px），通过CSS媒体查询与LayoutContext状态联动，实现了界面元素的自适应重排与功能组件的条件渲染。
