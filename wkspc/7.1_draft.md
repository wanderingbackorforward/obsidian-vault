# 第7章 系统实现

## 7.1 系统功能与平台架构

城市大型地下基础设施暗挖建造过程中，隧道围岩变形、地表沉降、裂缝扩展、环境振动等多源异构监测数据的实时采集与协同管控，对信息化平台的架构设计和工程化实现能力提出了较高要求。本章基于前文的需求分析与系统设计方案，从工程实现角度阐述隧道监测数字孪生平台的技术路线与核心模块开发过程。针对隧道施工现场数据来源分散、监测指标类型多样、管理层级差异显著等实际问题，本系统采用前后端分离的Browser/Server架构，前端基于React构建面向不同角色的交互界面，后端基于Flask提供数据服务与智能分析接口，数据层通过工厂模式支持云端与本地数据库的灵活切换，以适应隧道项目从原型验证到现场部署的不同阶段需求。全章围绕数据采集与融合、智能分析与人工智能（Artificial Intelligence，简称AI）决策、集群控制与远程管控三个方面，展开各功能模块的实现细节。

围绕隧道暗挖施工全过程的监测管控需求，系统将业务划分为四大功能域，共计15个功能模块，如表7-2所示。第一类为**变形与结构安全监测**，涵盖地表沉降监测、围岩裂缝监测、钢筋应力监测及InSAR卫星遥感（Interferometric Synthetic Aperture Radar，简称InSAR）地表形变分析四个模块，分别对应隧道施工中最为关键的竖向位移、结构损伤、受力状态和区域形变四类监测指标。第二类为**施工环境监测**，包括隧道内温湿度采集和施工振动监测两个模块，用于掌握洞内环境参数和爆破振动对周边建构筑物的影响。第三类为**施工过程管控**，包含盾构机轨迹追踪、施工工单管理、隧道断面管理和数据批量导入四个模块，支撑从掘进参数监控、问题闭环处理到断面级风险评估的施工管理流程。第四类为**智能分析与决策**，包括沉降预测分析、多指标异常检测、高级统计分析、AI智能助手和总览仪表盘五个模块，为工程师提供从趋势预判、异常预警到辅助决策的全链路智能分析能力。

系统总体架构采用表现层、业务逻辑层和数据持久层的三层结构，如图7-1所示。表现层面向隧道施工现场的工程师、项目管理人员和数据分析人员，使用React和TypeScript构建单页应用（Single Page Application，简称SPA），为上述15个功能模块分别设计了对应的操作界面，并采用懒加载策略——仅在用户访问特定监测功能时才加载对应页面，从而有效控制系统的初始加载时间。业务逻辑层基于Flask搭建，将15个功能模块各自注册为独立的服务单元，每个模块拥有独立的数据处理与接口逻辑，便于针对不同监测业务进行并行开发与独立维护。数据持久层的设计充分考虑了隧道项目的部署场景多样性：云端部署阶段使用Supabase托管的PostgreSQL数据库，通过REST应用程序接口（Application Programming Interface，简称API）进行访问；而企业内网或施工现场离线环境下，可通过配置切换至MySQL数据库，两种数据源对上层业务完全透明，切换时无需修改任何业务代码。

系统的技术选型以隧道工程监测场景的实际需求为导向。界面开发方面，选择React框架的组件化开发模式，使得沉降、裂缝、振动等各监测模块可以作为独立组件分别开发和维护，适合功能模块较多的综合监测平台。后端选择Python语言，主要原因在于隧道监测系统需要集成沉降预测、异常检测、因果推断等多种机器学习（Machine Learning，简称ML）模型，Python在科学计算与机器学习领域具有最为成熟的工具链支持，便于将数据分析算法与监测业务服务进行集成。数据可视化方面，针对隧道监测数据的时序特征和空间分布特点，系统集成了三类可视化引擎：（1）ECharts图表库用于绘制沉降曲线、温度场分布和振动频谱等时序监测图表；（2）Leaflet地图引擎用于InSAR卫星遥感地表形变和隧道线路走向的空间展示；（3）Three.js三维引擎用于隧道结构的数字孪生可视化，支持工程师从三维视角观察隧道线型与变形情况。各类技术选型及其在系统中的具体用途详见表7-1。

隧道监测系统面向的用户角色差异显著：项目管理员负责系统配置与用户管理，现场工程师承担数据录入与分析研判，而上级管理人员主要查阅监测报告与预警信息。针对这一多角色协同需求，系统实现了基于角色的访问控制（Role-Based Access Control，简称RBAC）机制。认证上下文模块中定义了管理员（admin）、工程师（user）和访客（guest）三级角色体系，其中管理员拥有用户管理、模块配置、数据删除等全部权限，工程师可使用数据采集、预测分析、工单处理等核心业务功能，访客仅具备数据浏览权限。用户登录后，前端将用户信息存储于全局认证状态中，后续请求自动携带身份标识。在路由层面，权限守卫（AuthGuard）组件负责拦截未认证或权限不足的访问请求，保障隧道监测数据的访问安全。

前端工程架构的一项重要设计是功能模块的动态管理机制。隧道监测平台的15个功能模块涉及从基础的沉降裂缝数据查看到高级的机器学习预测分析，各模块的开发进度和上线节奏存在差异——例如沉降监测模块在施工初期即需投入使用，而因果推断等高级分析功能可能在积累足够数据后才具备上线条件。为此，系统设计了模块开关控制机制：每个功能模块在后台配置中标记为"已上线"或"待上线"状态，系统在页面加载时自动检查模块状态，已上线模块正常使用，待上线模块则展示"功能开发中"的提示信息。项目管理员可在管理后台随时调整各模块的上线状态，系统记录每次变更的操作时间与操作人，形成审计追踪。此外，考虑到隧道施工现场手持终端与项目办公室大屏的显示差异，系统内置了三档屏幕适配方案（手机、平板和桌面），同一监测数据在不同设备上自动调整为适宜的展示布局。

后端服务的设计重点在于保障隧道施工现场复杂网络环境下系统的可用性。15个功能模块在后端各自拥有独立的数据处理逻辑，沉降分析模块负责断面沉降数据的入库与趋势计算，裂缝监测模块处理裂缝宽度时序数据与扩展速率分析，振动监测模块完成频谱解析与峰值提取，盾构遥测模块管理掘进参数与环级偏差计算，AI助手模块提供自然语言交互式数据查询——各模块独立开发、独立维护，便于根据施工阶段需求灵活扩展。针对隧道施工现场网络条件不稳定的实际问题，系统实现了三级服务保障机制：正常状态下，各监测页面直接请求后端获取实时数据；当网络波动导致请求连续失败时，系统自动判定后端暂时不可达，切换至本地缓存模式，现场工程师仍可查阅已缓存的监测数据与历史分析结果；经过冷却期后系统自动尝试恢复在线连接。对于沉降预测等机器学习推理接口，由于深度学习模型的计算耗时较长，系统为其设置了更宽裕的响应等待时间，避免因计算延迟而误判为服务中断。

系统部署在Vercel平台上，采用无服务器（Serverless）架构实现云端托管。部署配置中将函数最大执行时间设为60秒，主要考虑到隧道沉降预测等机器学习推理接口的计算耗时较长。同时配置了SPA路由重写规则，确保用户直接访问特定监测页面或刷新浏览器时路由系统能正常工作。针对隧道施工现场的移动巡检需求，系统通过Capacitor框架将Web应用封装为Android原生应用，实现了同一代码库同时服务于桌面端监控大屏与移动端现场巡检两种使用场景。移动端通过环境变量检测运行环境，自动切换为适配小屏幕的卡片式布局——桌面端多栏并列显示的监测数据面板在移动端转换为可滑动切换的单栏卡片。整个响应式方案覆盖移动端、平板和桌面三个断点区间，满足隧道工程从施工现场到项目办公室的多场景使用需求。
