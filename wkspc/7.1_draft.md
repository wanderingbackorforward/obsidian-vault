# 第7章 系统实现

## 7.1 系统功能与平台架构

城市大型地下基础设施暗挖建造过程中，隧道围岩变形、地表沉降、裂缝扩展、环境振动等多源异构监测数据的实时采集与协同管控，对信息化平台的架构设计和工程化实现能力提出了较高要求。本章基于前文的需求分析与系统设计方案，从工程实现角度阐述隧道监测数字孪生平台的技术路线与核心模块开发过程。针对隧道施工现场数据来源分散、监测指标类型多样、管理层级差异显著等实际问题，本系统采用前后端分离的Browser/Server架构，前端基于React构建面向不同角色的交互界面，后端基于Flask提供数据服务与智能分析接口，数据层通过工厂模式支持云端与本地数据库的灵活切换，以适应隧道项目从原型验证到现场部署的不同阶段需求。全章围绕数据采集与融合、智能分析与人工智能（Artificial Intelligence，简称AI）决策、集群控制与远程管控三个方面，展开各功能模块的实现细节。

系统总体架构采用表现层、业务逻辑层和数据持久层的三层结构，如图7-1所示。表现层面向隧道施工现场的工程师、项目管理人员和数据分析人员，使用React 18和TypeScript构建单页应用（Single Page Application，简称SPA），配合Vite作为构建工具实现模块热替换与快速打包。考虑到隧道监测系统功能模块较多且各监测页面包含大量图表组件，前端通过React Router v6管理涵盖沉降监测、裂缝监测、温度监测、振动监测、InSAR卫星遥感（Interferometric Synthetic Aperture Radar，简称InSAR）、盾构轨迹追踪、总览仪表盘等在内的15个核心页面路由，并对所有页面组件采用懒加载策略，仅在用户访问特定功能时才加载对应代码包，从而有效控制首屏加载体积。业务逻辑层基于Flask搭建，通过蓝图（Blueprint）机制将隧道监测涉及的各类业务拆分为15个功能模块——沉降分析、裂缝监测、温度采集、振动处理、InSAR数据解析、工单管理、盾构机遥测等各自独立注册路由前缀，实现了业务逻辑的解耦。数据持久层的设计充分考虑了隧道项目的部署场景多样性：云端部署阶段使用Supabase托管的PostgreSQL数据库，通过REST应用程序接口（Application Programming Interface，简称API）进行访问；而企业内网或施工现场离线环境下，可通过环境变量切换至MySQL数据库。两种数据访问实现遵循统一的Repository接口，上层业务代码无需关心底层数据源的差异。

系统的技术选型以隧道工程监测场景的实际需求为导向。前端选择React生态，其组件化开发模式适合功能模块较多的综合监测平台，TypeScript的静态类型检查有助于减少跨模块调用中的接口不一致问题。后端选择Python和Flask，主要原因在于隧道监测系统需要集成沉降预测、异常检测、因果推断等多种机器学习（Machine Learning，简称ML）模型，Python在科学计算与机器学习领域具有成熟的工具链支持，而Flask的轻量特性使得API开发与模型推理调用之间的集成较为简洁。数据可视化方面，针对隧道监测数据的时序特征和空间分布特点，系统集成了ECharts图表库用于绘制沉降曲线、温度场分布和振动频谱等监测图表，集成Leaflet地图引擎用于InSAR地表形变和隧道线路的空间展示，并通过Three.js实现隧道结构的三维数字孪生可视化。各类技术选型及其在系统中的具体用途详见表7-1。

隧道监测系统面向的用户角色差异显著：项目管理员负责系统配置与用户管理，现场工程师承担数据录入与分析研判，而上级管理人员主要查阅监测报告与预警信息。针对这一多角色协同需求，系统实现了基于角色的访问控制（Role-Based Access Control，简称RBAC）机制。认证上下文模块中定义了管理员（admin）、工程师（user）和访客（guest）三级角色体系，其中管理员拥有用户管理、模块配置、数据删除等全部权限，工程师可使用数据采集、预测分析、工单处理等核心业务功能，访客仅具备数据浏览权限。用户登录后，前端将用户信息存储于全局认证状态中，后续请求自动携带身份标识。在路由层面，权限守卫（AuthGuard）组件负责拦截未认证或权限不足的访问请求，保障隧道监测数据的访问安全。

前端工程架构的一项重要设计是动态模块管理机制。隧道监测平台的功能覆盖面广，从基础的沉降裂缝监测到高级的机器学习预测分析，各模块的成熟度和上线节奏存在差异。为此，系统设计了模块门控（ModuleGate）组件配合模块上下文（ModulesContext）实现功能模块的动态控制。每个功能模块在配置中标记为"已开发"（developed）或"待开发"（pending）状态，模块门控在页面渲染前进行状态检查：已开发模块正常加载，待开发模块则展示占位提示信息。管理员可在模块管理后台调整各模块的状态，系统记录每次变更的操作时间与操作人，形成审计追踪。状态管理方面，系统根据隧道监测数据的不同特点采用分层方案：（1）认证信息与模块配置等全局状态通过React Context管理；（2）布局适配通过响应式上下文管理，定义了移动端、平板和桌面三档断点，以适应施工现场手持设备与办公室大屏的不同使用场景；（3）AI智能助手的对话状态等高频更新数据采用Zustand状态库管理，设置30秒防抖机制以减少不必要的存储写入。

后端API服务的设计重点在于模块化组织与容错保障。Flask主服务通过蓝图机制注册了涵盖沉降、裂缝、温度、振动、InSAR、工单、盾构机遥测、高级分析、AI助手等业务的15个功能模块，各模块拥有独立的路由前缀和处理逻辑，便于团队针对不同监测业务并行开发。前端调用后端接口时，经过三层渐进式API客户端架构来保障服务可用性。基础层负责HTTP请求封装，遇到服务端错误时自动延迟重试。中间层在此基础上增加了熔断逻辑：当连续请求失败达到阈值时，系统判定后端暂时不可用，自动切换至本地缓存数据模式，经过冷却期后再尝试恢复连接。最上层为机器学习接口设置了更长的超时时间，以适应深度学习模型推理的计算耗时。这种渐进式降级设计对于隧道施工现场网络条件不稳定的环境尤为关键，确保即使后端服务短暂中断，现场工程师仍能查阅已缓存的监测数据与历史分析结果。

系统部署在Vercel平台上，采用无服务器（Serverless）架构实现云端托管。部署配置中将函数最大执行时间设为60秒，主要考虑到隧道沉降预测等机器学习推理接口的计算耗时较长。同时配置了SPA路由重写规则，确保用户直接访问特定监测页面或刷新浏览器时路由系统能正常工作。针对隧道施工现场的移动巡检需求，系统通过Capacitor框架将Web应用封装为Android原生应用，实现了同一代码库同时服务于桌面端监控大屏与移动端现场巡检两种使用场景。移动端通过环境变量检测运行环境，自动切换为适配小屏幕的卡片式布局——桌面端多栏并列显示的监测数据面板在移动端转换为可滑动切换的单栏卡片。整个响应式方案覆盖移动端、平板和桌面三个断点区间，满足隧道工程从施工现场到项目办公室的多场景使用需求。
