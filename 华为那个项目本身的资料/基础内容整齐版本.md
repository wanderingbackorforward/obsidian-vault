# 华为实习项目梳理版

## 1. 项目定位

这个项目是一个华为内部的**文档扫描与文档质量检查系统**。

它并不是一个典型的纯 ReAct Agent 项目，而是一个**pipeline 为主体，部分环节引入类 Agent / LLM 二次校验能力**的工程系统。

整体技术栈主要包括：

- FastAPI
    
- MySQL
    
- 线程池
    
- 多类型文档解析
    
- LLM common check
    
- LLM / 类 Agent 二次校验
    

一句话概括：

> 这是一个面向内部文档质量检查的自动化扫描系统，整体以 pipeline 流程为主，通过多文档解析、chunk 化、LLM 高召回检查和二次证据校验，实现对文档中语法、格式、链接、安全合规等问题的识别。

---

## 2. 系统整体工作流程

系统整体流程可以分成两层：**任务调度层**和**文档检查执行层**。

### 2.1 任务调度层

首先，前端通过 FastAPI 发起文档扫描任务请求。

后端接收到任务后，不会同步阻塞执行，而是将任务放入线程池中异步处理。

同时，系统会在 MySQL 中维护任务状态，主要通过：

- task 表
    
- 文档任务表
    

来记录每个任务、每个文档的执行进度和检查结果。

前端则通过查询 MySQL 中的任务状态，获取当前扫描进度和最终结果。

这一层主要解决的是：

> 多文档扫描任务的异步执行、状态管理和前后端进度反馈问题。

---

### 2.2 文档检查执行层

对于具体的文档检查流程，系统会先对不同类型的文档进行解析和 chunk 化。

虽然不同文档格式差异很大，但最终都会被处理成统一的 chunk 结构，方便后续交给大模型进行检查。

大致流程是：

1. 根据文档类型进行解析；
    
2. 将文档内容切分成 chunk；
    
3. 对 chunk 执行一次 common check；
    
4. common check 追求高召回，先尽可能找出疑似问题；
    
5. 对 common check 返回的问题进行二次筛选；
    
6. 只有有明显证据的问题，才进入最终反馈环节。
    

common check 返回的信息主要包括：

- 正文内容
    
- 问题类型
    
- 修改意见
    
- 相关上下文
    

这里的关键设计是：

> 第一次检查追求高召回，宁可多发现疑似问题；第二次校验追求准确性，只把证据明确的问题反馈给用户。

---

## 3. 多类型文档 chunk 化

不同类型文档的 chunk 化方式不同。

例如 PPT 相对更天然一些，可以按照文本框进行提取。

但 PDF 更复杂，因为 PDF 本身不是一种天然结构化的文档格式。它更像是一个排版结果，文本和链接都可能是分散存储的。

PDF 的处理流程大致是：

1. 先提取页面中的文本块；
    
2. 根据文本块的上下位置，将相近内容合并成一行；
    
3. 再根据缩进、行距、位置关系进行段落拼接；
    
4. 最后形成可以给 LLM 检查的 chunk。
    

也就是说，PDF 不能简单地按照“自然段”读取，而是需要自己基于坐标和布局信息还原段落结构。

---

## 4. 我参与的核心工作：多文档链接提取

我参与的一个重点模块是：**文档 chunk 化过程中的多类型文档链接提取**。

这里最核心的问题是：

> 对于 PDF，链接和具体文字很难完全精确对齐。

原因是 PDF 本身并不是结构化格式。

在 PDF 中，链接通常是基于坐标区域存在的；而文本内容则是分散存储的。也就是说，系统能拿到一个链接的坐标区域，也能拿到很多文本块的位置，但很难天然知道“这个链接到底对应哪几个字”。

所以这里不能简单认为：

> 文本里出现了 URL，就一定能对应到 PDF 中的 link。

也不能简单认为：

> 某个 link 坐标范围内的文字，就一定完整构成这个链接的上下文。

因为文本块可能被拆散，链接区域也可能覆盖不完整。

---

## 5. PDF 链接提取的工程方案

由于 PDF 链接和文本无法做到完全精准对齐，所以我们采取了一个工程上的折中方案。

核心思路是：

> 不追求字级别的精确绑定，而是降低要求，把链接绑定到更稳定的段落级 chunk 上。

具体做法是：

1. 先对 PDF 文本进行行合并和段落整合；
    
2. 得到带有位置信息的段落；
    
3. 对于每一个处理好的段落，在同一页中找到距离它最近的 link；
    
4. 将这个 link 先绑定到当前段落；
    
5. 如果后续 chunk 需要跨页拼接，再在段落已经带 link 的基础上进行跨页 chunk 拼接。
    

也就是：

> 先进行 link 和段落的拼接，再进行跨页 chunk 拼接。

这样做的好处是：

- 避免了直接在跨页 chunk 之后再找 link 导致位置关系丢失；
    
- 保留了 PDF 页面内 link 和文本的相对空间关系；
    
- 虽然不是字级别精准，但在工程上更稳定、更可控；
    
- 后续 LLM 检查时也能拿到链接及其附近上下文。
    

---

## 6. 系统识别的问题类型

系统主要识别的文档问题包括：

1. 标点符号问题；
    
2. 大小写问题；
    
3. 语言语法问题；
    
4. 链接内容验证；
    
5. 内容合规安全；
    
6. 账号密码等敏感信息泄露；
    
7. 不规范用语和错误用语。
    

其中，标点符号、大小写、基础语法类问题，目前主要可以通过通用 common check 加二次校验处理。

真正需要特殊设计的主要是三类：

1. 链接与内容验证；
    
2. 内容合规安全；
    
3. 不规范用语和错误用语。
    

---

# 三类重点问题的处理方式

## 7. 不规范用语 / 错误用语问题

这个问题内部可以对应到类似 5005 这类不规范用语问题。

它的核心难点是：**误报非常多**。

尤其是内部文档中存在大量内部术语、项目代号、产品名、缩写词，这些词很容易被大模型误判为“不规范用语”或者“错误用语”。

---

### 7.1 v1 方案：Prompt 白名单 + 规则引擎

最早的 v1 方案是：

1. 在 prompt 中编码一部分白名单；
    
2. 让 LLM 知道某些内部词汇不要误报；
    
3. 如果模型能力较弱，就额外通过 Python 规则引擎进行限制；
    
4. 用户也可以手动输入自己认为合理的白名单。
    

这个方案在一些场景下有效，但仍然存在问题。

最大的问题是：

> 新词、新术语会不断出现，系统很难提前把所有内部术语都写进白名单。

如果一个词不在白名单中，系统就会陷入两难：

- 直接报错，可能造成大面积误报；
    
- 全部标记为待确认，又会导致检查结果没有价值。
    

所以这里不能只靠静态白名单。

---

### 7.2 v2 方案：证据构建 + 打分机制 + 灰色区域 LLM 二次校验

针对 v1 的问题，我倾向于的改进方案是：

> 对疑似不规范用语构建证据，并引入打分机制；对于灰色区域的问题，再交给 LLM 做二次校验。

具体流程是：

1. common check 先筛出大量疑似不规范用语；
    
2. 规则引擎先做一轮过滤；
    
3. 对剩下的问题，扩展结构化字段；
    
4. 在全文或多文档中找到所有包含同名词的 chunk；
    
5. 对这些 chunk 构建证据并打分；
    
6. 根据证据强弱判断是直接报错、提醒加入白名单，还是进入 LLM 二次判断。
    

打分依据包括：

- 这个词在同一文档中的出现频率；
    
- 这个词在不同文档中的出现频率；
    
- 是否多次出现在标题中；
    
- 是否出现在定义句中；
    
- 是否有解释性上下文；
    
- 是否和已有白名单或内部术语相似。
    

例如，如果一个词：

- 在多个文档中反复出现；
    
- 多次出现在标题；
    
- 在某个地方还有“XX 是指……”这种定义句；
    

那么它很可能不是错误用语，而是一个新的内部术语。

这种情况下不应该直接报错，而应该提示：

> 疑似新术语，建议人工确认后加入白名单。

如果证据明显不足，且模型和规则都认为它是错误表达，那么才进入报错。

这个方案的本质是：

> 不再让 LLM 只根据单个 chunk 做孤立判断，而是给它补充跨 chunk、跨文档的证据，让它基于证据判断这个词到底是错误用语，还是可能的新术语。

---

## 8. 疑似密码 / 敏感信息泄露问题

对于疑似账号密码泄露这类问题，目前处理方式不需要特别复杂。

主要是通过 prompt 明确判断标准，让模型结合上下文区分几种情况：

1. 示例代码；
    
2. 占位符；
    
3. 明显真实泄露。
    

例如：

### 示例代码

如果文档中出现的是示例代码，需要提醒模型注意：

> 示例代码中的账号密码可能被用户复制后直接用于实验，因此也可能存在风险，需要给出警告。

### 占位符

如果是明显的占位符，比如：

- `<password>`
    
- `your_token_here`
    
- `xxxxxx`
    
- `******`
    

这类一般可以通过，不作为真实泄露处理。

### 明显泄露

如果出现了看起来像真实账号、真实密码、真实 token、真实密钥的内容，则直接报错。

这一类问题相比链接验证和术语误报，复杂度低一些，主要依赖上下文判断和 prompt 约束。

---

## 9. URL / 链接验证问题

URL 验证是最复杂的一类问题。

它的问题在于：**链接是否有效，不能只看 HTTP 状态码。**

常见复杂情况包括：

1. 锚点链接不应该作为普通外链检查；
    
2. 403 不一定代表链接失效，可能只是服务端反爬；
    
3. 请求缺少浏览器请求头时，可能被错误拦截；
    
4. 返回 200 也不一定正确，可能是软 404；
    
5. 返回 200 也可能是广告页或恶意劫持页；
    
6. 链接可能跳转到首页兜底；
    
7. 链接可能跳转到登录页；
    
8. 旧链接可能没有 301，只是在页面里提示迁移到新地址；
    
9. 下载链接可能返回 200，但实际返回的是 HTML 错误页；
    
10. 页面可能需要 JS 渲染，静态 HTML 只有空壳。
    

所以 URL 验证不能简单写成：

```python
if status_code == 200:
    return "可用"
else:
    return "不可用"
```

这会导致大量误判。

---

## 10. URL 验证的类 ReAct 循环

面对 URL 验证中大量复杂分支，我们没有选择用 if-else 把所有情况写死。

而是提供一组工具，让大模型根据 observation 选择下一步操作，形成一个类 ReAct 的循环。

整体思想是：

> 系统只保留必要的硬规则兜底，复杂判断交给 LLM 根据观察结果和工具调用进行推理。

if-else 主要只放在两处。

---

### 10.1 第一处 if-else：进入 ReAct 前的硬性兜底

在进入类 ReAct 循环前，系统会先处理一些明确规则。

例如：

- 锚点链接；
    
- 明确的 401；
    
- 明显不需要检查的内部片段；
    
- 明确格式非法的 URL。
    

这些情况不需要 LLM 反复推理，直接由规则处理。

---

### 10.2 第二处 if-else：循环内的工具调度

第二处 if-else 是工具调用层面的调度。

也就是说，系统需要根据 LLM 选择的 tool name，调用对应工具。

但“下一步该用哪个工具”不是由我们写死，而是由 prompt 引导模型根据 observation 选择。

---

## 11. URL 验证循环中的工具类型

循环内部的 function 大体分成三类。

### 第一类：继续取证类工具

用于获得更多证据，例如：

- 重试请求；
    
- 跟随跳转链接；
    
- 探测浏览器请求头；
    
- 模拟浏览器访问；
    
- 获取最终落地页；
    
- 检查资源类型。
    

这一类工具主要解决：

> 当前 observation 不足以判断，需要继续访问或补充证据。

---

### 第二类：抽取结构化信息类工具

用于从页面或响应中提取结构化信息，例如：

- 页面标题；
    
- 正文摘要；
    
- 错误提示；
    
- 迁移提示；
    
- 登录提示；
    
- 下载资源类型；
    
- 关键 meta 信息。
    

这一类工具主要解决：

> 页面返回了内容，但需要从内容里提取判断依据。

---

### 第三类：判断匹配类工具

用于判断链接结果是否和原文上下文匹配，例如：

- 链接语义匹配；
    
- 页面内容与文档上下文匹配；
    
- 判断是否跳到了首页；
    
- 判断是否是软 404；
    
- 判断是否是登录页；
    
- 判断资源类型是否符合预期。
    

这一类工具主要解决：

> 链接虽然可访问，但访问到的内容是不是文档里真正想引用的内容。

---

## 12. URL 验证的最终输出

类 ReAct 循环最终输出三类结果：

1. 存在；
    
2. 不存在；
    
3. 模糊。
    

其中“模糊”很重要。

因为有些情况系统确实无法稳定判断，例如：

- 页面必须登录；
    
- 页面依赖 JS 渲染；
    
- 页面有反爬；
    
- 页面返回信息不足；
    
- 页面提示迁移，但没有明确新地址。
    

这些场景如果强行判断成存在或不存在，都会带来误报。

所以系统会保留“模糊”状态，用于后续人工确认或低置信度提示。

---

## 13. Prompt 设计方式

URL 验证这里的 prompt 需要明确告诉模型：这是一个类 ReAct 的判断流程。

Prompt 中要包含几类信息：

第一，明确当前任务：

> 你需要判断这个 URL 在当前文档上下文中是否有效，而不是只判断 HTTP 状态码。

第二，要求模型先判断当前属于哪一种失败模式。

第三，明确列出常见 failure mode。

第四，解释不同工具的含义：

- 哪些工具是继续取证；
    
- 哪些工具是抽取结构化信息；
    
- 哪些工具是判断匹配。
    

第五，要求模型根据 observation 选择下一步工具，而不是一次性武断输出。

---

## 14. URL 验证中的有限 failure mode

虽然 URL 的异常情况看起来很多，但可以抽象成有限几类 failure mode：

1. 明确不可达；
    
2. 访问受限；
    
3. 中间页跳转；
    
4. 资源迁移页，也就是页面提示新地址；
    
5. JS 渲染导致静态内容不足；
    
6. 资源类型不匹配，例如预期是文件，结果返回 HTML；
    
7. 上下文不匹配，例如跳到了首页、广告页、登录页或无关页面。
    

这个抽象很关键。

因为如果不抽象 failure mode，就会变成无穷无尽的 if-else。

但抽象之后，系统只需要围绕这些 failure mode 设计工具和 prompt，剩下的复杂情况交给模型根据 observation 来判断。

---

## 15. 一些典型复杂 case

实际可能遇到的奇怪情况包括：

1. 短链或跟踪链接最终跳转到错误资源；
    
2. 下载链接返回 200，但实际内容是 HTML 错误页，不是真正的文件；
    
3. 页面返回 200，但内容依赖 JS 渲染，静态 HTML 只有空壳；
    
4. 旧文档不做 301 跳转，只在页面里提示“迁移到新版”；
    
5. 页面跳转到首页兜底，但原目标资源其实不存在；
    
6. 页面跳转到登录页，无法判断登录后内容是否存在；
    
7. 访问时缺少浏览器请求头，导致服务端返回 403；
    
8. 页面存在，但和文档上下文引用的内容不匹配。
    

这些情况说明：

> URL 验证不是一个简单的请求状态码问题，而是一个结合网络访问、页面内容、跳转链路和上下文语义的综合判断问题。

---

## 16. 关于 LLM 调用频率

这个类 ReAct 流程确实可能带来一个问题：频繁调用 LLM。

但实际可以通过分流机制控制成本。

并不是所有 URL 都需要进入完整 ReAct 循环。

大部分简单问题可以一次性完成，例如：

- 明显锚点链接；
    
- 明显格式错误；
    
- 明确跳转到首页；
    
- 明显伪 404；
    
- 明确访问失败；
    
- 明确不需要检查的内部链接。
    

只有复杂、证据不足、需要多步取证的问题，才进入类 ReAct 循环。

所以整体策略是：

> 简单问题规则和一次性判断解决，复杂问题才进入 LLM 工具循环。

# 面试时的压缩表达版

如果面试官问“你这个项目是怎么做的”，可以这样讲：

> 这个项目是华为内部的文档质量扫描系统，不是一个纯 ReAct Agent，而是一个 pipeline 为主体的工程系统。后端用 FastAPI 接收任务，通过线程池异步执行，并用 MySQL 的 task 表和文档任务表维护进度，前端通过查询任务状态展示扫描进度。
> 
> 具体检查流程是，先对不同类型文档做解析和 chunk 化，然后让大模型做一次 common check，先保证高召回，把疑似问题找出来。之后再做二次校验，只有有明确证据的问题才进入最终反馈。
> 
> 我参与比较多的是多文档链接提取和部分二次校验策略。比如 PDF 里链接和文字很难精确对齐，因为 PDF 的链接是坐标区域，文本又是分散存储的。所以我们没有强行做字级别绑定，而是先根据位置做行合并和段落拼接，再把同页最近的 link 绑定到段落上，最后再做跨页 chunk 拼接。这样虽然是工程折中，但稳定性更好，也能给后续 LLM 提供比较完整的链接上下文。
> 
> 除了链接提取，系统还会检查语法、标点、链接有效性、内容合规、安全泄露和不规范用语。其中 URL 验证比较复杂，不能只看 200 或 404，因为会遇到软 404、403 反爬、跳登录页、首页兜底、资源迁移、JS 渲染空壳等情况。所以我们没有把所有情况写成 if-else，而是抽象出有限 failure mode，再提供工具给大模型做类 ReAct 的取证循环，最终输出存在、不存在或模糊。
> 
> 对于不规范用语，我们也遇到过大量误报，因为内部术语很容易被模型误判。所以后续设计上不是单纯靠 prompt 白名单，而是构建证据和打分机制，比如看这个词是否多次出现、是否跨文档出现、是否出现在标题或定义句里。证据强的可能提示为新术语，建议加入白名单；证据弱且模型判断明确的，才作为问题反馈。

# 更短的 30 秒版本

> 我这个华为项目是一个内部文档质量扫描系统，整体不是纯 ReAct Agent，而是 pipeline 为主体。FastAPI 接任务，线程池异步执行，MySQL 维护任务和文档进度。文档会先按类型解析成统一 chunk，再用 LLM 做 common check 高召回检查，之后通过规则和二次校验过滤误报。
> 
> 我主要参与的是多文档链接提取和部分复杂问题的校验策略。PDF 链接提取比较难，因为 PDF 里链接是坐标区域，文本是分散存储的，没法天然对齐。所以我们先做文本行合并和段落拼接，再把同页最近的 link 绑定到段落，最后再跨页拼 chunk。
> 
> 另外 URL 验证不能只看状态码，因为会遇到软 404、403 反爬、登录页、首页兜底、资源迁移等情况，所以我们抽象 failure mode，让 LLM 通过类 ReAct 的工具循环继续取证，最后判断存在、不存在或模糊。

# 这个项目最值得你讲的亮点

我建议你面试时重点讲这三个：

1. **PDF 解析和链接对齐的工程折中**  
    这个很具体，也能体现你理解文档解析的复杂性。
    
2. **common check 高召回 + 二次校验高精度**  
    这个能说明你不是简单调用 LLM，而是在设计完整质量控制流程。
    
3. **URL 验证的有限 failure mode + 类 ReAct 工具循环**  
    这个最像 AI 工程亮点，但不要说成纯 Agent，要说成“pipeline 中针对复杂链接验证场景设计的类 ReAct 取证流程”。
    

最稳的项目定位是：

> 这是一个以工程 pipeline 为主体、在复杂判断环节引入 LLM / 类 Agent 校验能力的文档质量检查系统。
