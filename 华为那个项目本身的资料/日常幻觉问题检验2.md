这个问题和“连接/连结”那类不完全一样。

这里更像是：

> **英文扫描规则边界没有讲清楚，模型把“正常英文写法 / 不可见空格 / 祈使句”当成错误了。**

群里公告说：

> 优化英文文档语法错误扫描和空格错误扫描，增加空格约束和祈使句

这个信息其实已经能反推出大概怎么修的。

# 1. `xxxxx.` 这个空格误报到底可能是什么？

你看到的是：

```text
xxxxx. 
```

模型报：

> 句尾的句号前边不应该有空格，请删除

但实际看起来像是句号**后面**有空格，而不是句号**前面**。

这里有两种可能。

---

## 可能一：模型把“句号后的空格”误判成“句号前的空格”

英文里：

```text
This is a sentence. Next sentence.
```

句号后面有一个空格是正常的。

如果文本是：

```text
xxxxx. 
```

这可能只是句尾后有一个空格，严格说它是 trailing space，但不一定是“英文语法错误”。

如果 prompt 只说：

> 检查英文标点和空格错误

模型可能会很泛地认为：

> 这里有空格问题。

然后在描述时套用了常见模板：

> 句号前不应有空格。

于是就出现了：

> 实际是句号后空格，但建议写成句号前空格。

这本质是**空格规则没有精确定义**。

---

## 可能二：空格在 prompt 里不可见，模型无法稳定定位

空格、换行、制表符这类东西对 LLM 很不友好。

比如：

```text
xxxxx. 
```

和

```text
xxxxx.
```

肉眼都很难看出来区别。

如果 chunk 经过 Markdown、JSON、HTML、日志打印、前端展示，很可能空格被压缩、转义或丢失。

于是模型只能“猜”这个空格在哪里。

所以它可能把：

> 句号后的 trailing space

说成：

> 句号前有空格。

这不是模型不会英语，而是**空白字符定位不适合纯自然语言判断**。

---

# 2. “增加空格约束”大概率修了什么？

我猜他们做的不是换模型，而是加了类似下面的约束。

原来可能只是说：

> 检查英文文档中的空格错误。

这太宽泛了。

后面应该改成了：

> 只有以下情况才算空格错误：
> 
> 1. 标点符号前存在多余空格，例如 `word .`
>     
> 2. 单词之间有连续多个空格，例如 `word word`
>     
> 3. 括号、冒号、逗号等标点周围不符合英文规范
>     
> 4. 不要把句号后正常的单个空格当作错误
>     
> 5. 如果无法明确定位空格位置，不要报错
>     

特别是这一条最关键：

> **不要把句号后正常的单个空格当成错误。**

也就是说，模型以前可能只知道“空格异常要报”，但不知道哪些空格是允许的。

# 3. 空格问题最稳的做法其实不是靠 LLM

如果是我改，我会把英文空格错误拆出来，用 Python 规则先扫。

比如：

```python
import re

def check_english_spacing(text: str):
    issues = []

    # 标点前多余空格：word . / word , / word :
    for m in re.finditer(r"\w+\s+[.,;:!?]", text):
        issues.append({
            "type": "space_before_punctuation",
            "error_text": m.group(),
            "suggestion": re.sub(r"\s+([.,;:!?])", r"\1", m.group())
        })

    # 单词之间多个空格
    for m in re.finditer(r"\b\w+\s{2,}\w+\b", text):
        issues.append({
            "type": "multiple_spaces_between_words",
            "error_text": m.group(),
            "suggestion": re.sub(r"\s{2,}", " ", m.group())
        })

    return issues
```

然后 LLM 只负责解释：

> 为什么这里不应该有空格。

而不是让 LLM 自己判断空格在哪里。

因为空格问题本质是**确定性文本规则**，不是语义推理问题。

# 4. “增加祈使句”又是什么意思？

这个也很关键。

英文技术文档里大量句子是祈使句，例如：

```text
Click OK.
Enter the password.
Select the target device.
Restart the service.
Run the following command.
```

这些句子没有显式主语，但它们不是语法错误。

因为祈使句默认主语是 `you`。

如果 prompt 只是让模型检查英文语法，模型可能会把这些句子误判成：

> 缺少主语  
> 句子不完整  
> 建议改成 You should click OK.

这在技术文档里反而是不合适的。

所以“增加祈使句”大概率是告诉模型：

> 在英文操作手册、技术文档、步骤说明中，以动词原形开头的命令式句子是合法的，不要因为缺少主语而报错。

---

# 5. 所以群里那个优化为什么能解决？

因为他们大概率补了两个边界。

## 第一，空格边界

以前模型看到空格就可能泛化报错。

优化后明确：

> 哪些空格算错，哪些空格不算错。

尤其是：

```text
word .
```

才是句号前空格错误。

而：

```text
word. Next
```

句号后一个空格是正常英文。

---

## 第二，英文语法边界

以前模型可能把技术文档里的祈使句当成残句。

优化后明确：

> 技术文档中的命令式表达是合法句式。

比如：

```text
Click Submit.
Configure the network.
Check the log file.
```

不应该报缺主语。

---

# 6. 这个问题和前面的“无中生有”有什么区别？

前面的“连接/连结”是：

> 修改建议没有和原文 error span 绑定。

英文空格这个更像是：

> 规则定义太宽，导致模型把正常情况也纳入错误范围。

一个是 **grounding 问题**。

一个是 **判定边界问题**。

当然它们也有交集：  
如果模型说“句号前有空格”，也应该要求它输出 exact span。

比如：

```json
{
  "error_type": "space_before_punctuation",
  "error_text": "word .",
  "correct_text": "word.",
  "evidence": "word .",
  "position": "before_period"
}
```

如果原文里没有 `word .`，就不能报。

# 7. 面试里可以怎么讲？

你可以这样说：

> 英文模式下我们遇到过一些空格和语法误报，比如句号后正常空格被误描述成句号前空格，或者技术文档里的祈使句被当成语法不完整。后来内部优化方向是给英文扫描增加更明确的空格约束和祈使句规则。
> 
> 我理解这里的核心不是模型能力问题，而是英文文档检查的规则边界需要明确。比如句号前空格、连续多个空格可以作为错误，但句号后正常单空格不应该报；技术文档中 `Click OK`、`Run the command` 这种祈使句也是合法表达，不应该因为缺少主语而误报。
> 
> 如果让我继续优化，我会把空格类问题尽量规则化，用正则定位 exact span，再让 LLM 负责解释；语法类问题则在 prompt 中明确豁免祈使句、标题、列表项、代码片段等技术文档常见结构。

最短总结就是：

> **英文空格误报是规则边界不清，祈使句误报是英文技术文档语法场景没定义清楚。**
