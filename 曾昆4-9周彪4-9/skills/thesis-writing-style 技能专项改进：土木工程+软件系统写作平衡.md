```markdown
## 🎯 土木工程论文中的软件系统介绍（特别重要）

### 核心定位
本论文是**土木工程专业论文**，软件系统是研究工具和成果，而非研究主体。

### 写作原则

#### 1. 视角定位原则

✅ **正确的视角**：
- 从土木工程问题出发
- 软件是解决土木问题的工具
- 技术服务于土木应用

❌ **错误的视角**：
- 直接介绍软件架构和功能
- 把软件当作产品推广
- 过度强调技术先进性

**示例对比**：
```
❌ 错误："本系统采用微服务架构，实现了前后端分离，提高了系统的可扩展性..."
✅ 正确："针对大型桥梁结构健康监测的数据规模和实时性需求，系统采用分布式架构，确保能够..."
```

#### 2. 内容比例原则

在每个技术章节（如7.1-7.4）中，保持以下比例：

| 内容类型 | 占比 | 说明 |
|---------|------|------|
| 土木工程问题/需求 | 30-40% | 引出为什么需要这个功能 |
| 软件解决方案概述 | 20-30% | 怎么解决，大方向 |
| 适度技术细节 | 15-25% | 关键技术，点到为止 |
| 应用效果/土木价值 | 20-30% | 效果如何，对土木有何价值 |

**检查方法**：统计每节中各类段落的数量，确保比例合理。

#### 3. 技术细节深度控制

**可以写的技术细节**（1-2句话说明即可）：
- 架构设计的大方向（分布式、模块化等）
- 关键算法的原理（不涉及公式和代码）
- 数据处理流程（框图+简述）
- 技术选型的理由（为什么选这个技术）

**不能写的技术细节**：
- 具体代码实现
- 详细的数据库设计
- API接口定义
- 前端组件结构
- 开发过程和测试细节
- 纯技术性能对比

**示例**：
```
✅ 可以写：
"数据采集模块采用消息队列技术，确保高频传感器数据的可靠传输和缓冲，满足实时监测的需求。"
（一句话说清楚：用什么技术、解决什么问题）

❌ 不能写：
"数据采集模块基于RabbitMQ实现消息队列，配置了3个交换机和5个队列，采用发布-订阅模式，消息持久化采用..."
（过度细节，像软件设计文档）
```

#### 4. 论述链条完整性原则

每个技术点必须遵循完整链条：

```
土木问题 → 技术需求 → 解决思路 → 软件实现 → 应用效果
   ↑                                              ↓
   └──────────────── 始终围绕土木 ─────────────────┘
```

**检查清单**：
- [ ] 介绍任何软件功能前，是否先说明土木工程需求？
- [ ] 技术实现是否与土木应用场景绑定？
- [ ] 是否回到土木工程效果和价值？
- [ ] 整个链条是否流畅？

#### 5. 术语平衡原则

**土木术语应占主导**（约60-70%）：
- 结构、监测、传感器、应力、变形、健康状态、安全评估...

**软件术语适度使用**（约30-40%）：
- 系统、模块、数据、算法、分析、可视化、平台...

**混合表达**（推荐模式）：
- "监测数据采集系统"（土木+软件）
- "结构分析算法"（土木+软件）
- "安全评估平台"（土木+软件）
- "基于XXX技术的桥梁健康监测方案"（技术+土木）

#### 6. 章节结构模板

**推荐的章节结构**（基于软件+土木模板）：

```
7.X 章节标题（如"系统功能与平台架构"）

【引言段】(1段，100-150字)
- 说明本节在整个系统中的定位
- 从土木工程需求引出本节主题

【土木需求分析】(1-2段，每段150-200字)
- 详细阐述土木工程面临的具体问题
- 说明为什么需要这个功能/模块
- 提出技术需求

【软件设计方案】(2-3段，每段150-200字)
- 说明总体设计思路（从土木需求到软件方案的映射）
- 介绍主要功能模块（每个模块都与土木需求对应）
- 说明架构设计（适度技术细节，1-2句话）

【关键技术实现】(1-2段，每段100-150字)
- 重点技术的简要说明（不超过一段一个技术点）
- 强调技术如何解决土木问题
- 避免过度细节

【应用效果】(1段，100-150字)
- 说明系统在实际工程中的应用
- 强调对土木工程的价值
- 可包含性能指标（与土木需求相关的）

【本节小结】(可选，50-100字)
- 总结本节核心内容
```

#### 7. 禁止的写作方式（针对软件介绍）

**❌ 绝对禁止**：
- 产品宣传式语言："本系统处于国际领先水平..."
- 纯技术对比："相比XX系统，本系统采用了更先进的..."
- 软工文档式描述："用户管理模块包含登录、注册、权限管理..."
- 开发过程描述："在需求分析阶段，我们调研了..."
- 过度的技术细节："使用React 18.2版本，配合Redux状态管理..."
- 脱离土木场景："系统具有良好的可扩展性和可维护性..."

**✅ 正确的写作方式**：
- 需求驱动："针对大跨度桥梁的多点位监测需求，系统设计了..."
- 土木视角："为实现结构健康状态的实时评估，系统集成了..."
- 适度技术："采用分布式架构处理海量监测数据，确保..."
- 应用导向："系统在XX桥梁的应用表明，能够有效..."
```
